# FR-Harness 前十项补救设计

## 1. 背景与目标

本轮工作以课程 A 任务与通用要求为验收基准，补齐当前仓库中已经确认的十项差距，并形成可复查的代码、测试、文档、Git、GitHub Actions、评审、PR、CI 与容器镜像证据。

本轮遵循以下边界：

- 不伪造历史上未发生的 brainstorming、subagent、评审、提交或 CI 记录。
- 可以从现存会话、仓库和 Git 历史中回顾性整理事实，但必须明确标注“回顾性整理”。
- 正式项目文档继续保留在仓库根目录。
- 本轮新生成的过程性 Markdown 文件全部放在 `temp/`。
- 保留用户现有且未跟踪的 `A任务完成指南.md`，不擅自纳入提交。

## 2. 方案选择

采用“评分项驱动补齐”方案：每一项课程要求都尽量对应到实现、测试、文档和可验证证据。与最小补丁相比，该方案会增加评审和交付链工作；与整体重构相比，它控制改动范围，避免为追求结构统一引入不必要的回归风险。

安全凭据采用用户批准的 `keyring + 环境变量回退` 方案。

## 3. 凭据生命周期

新增独立的凭据存储模块，并通过接口隔离具体后端，便于测试时注入内存实现。

命令行提供：

- `credential set`：使用隐藏输入写入系统密钥库。
- `credential status`：只显示“已配置/未配置”和来源，不显示密钥。
- `credential update`：使用隐藏输入替换已有密钥。
- `credential clear`：删除系统密钥库中的密钥。

运行时解析顺序：

1. `OPENAI_API_KEY` 环境变量；
2. 系统 `keyring`；
3. 两者都没有时，交互式终端给出安全录入引导，非交互式环境返回可操作的错误信息。

Docker 和 CI 继续使用环境变量，不要求容器内存在桌面密钥库。日志、异常和状态命令不得输出密钥内容。

## 4. 声明式 Agent 配置

将空的 `config.py` 实现为声明式配置入口，覆盖至少以下参数：

- 最大迭代次数；
- 记忆/历史长度限制；
- 自动批准的低风险动作；
- 需要人工批准的高风险动作；
- 反馈摘要长度或相近的运行边界。

配置具有安全默认值，并允许从配置文件和环境变量覆盖。现有 API 尽量保持兼容；需要扩展的函数采用可选参数，避免破坏已有调用方。

空模块按职责处理：

- `config.py` 实质实现；
- `actions.py` 若能承载动作类型和校验则实现，否则删除并同步文档；
- 空的 `__init__.py` 作为合法包标记保留，不将其误判为实现漂移；
- SPEC 中不存在的模块或已迁移职责必须与实际代码对齐。

## 5. 规格与过程文档

`SPEC.md` 按课程评分结构补齐：

- 十项要求结构；
- 至少五个用户故事；
- 功能要求和非功能要求；
- 数据与接口约束；
- 威胁模型；
- A 类 Agent Harness 机制说明；
- 验收标准及其测试映射。

`SPEC_PROCESS.md` 增加规格形成与修订过程。可验证的用户选择和仓库变更按时间线记录；历史对话中无法恢复的精确措辞不补写成虚构逐字稿。当前补救设计过程将作为一轮有完整证据的新迭代加入。

`PLAN.md`、`README.md`、`AGENT_LOG.md` 与实现保持一致：

- Task 4–12 填写 Git 中已经存在的真实 commit hash；
- 记录本轮实际使用的 prompt、OpenCode 输出、人工判断和修复；
- 对未发生的历史 subagent 流程如实说明偏差；
- 建立“要求—实现—测试—证据”追踪关系。

## 6. 测试与安全验证

采用测试先行方式扩展覆盖：

- 使用假的凭据存储后端测试 set/status/update/clear；
- 验证密钥不会出现在 stdout、stderr、日志或异常中；
- 验证环境变量与 keyring 的优先级；
- 验证非交互式缺少凭据时的失败行为；
- 验证配置默认值、覆盖方式和非法配置；
- 保持现有单元测试、演示和容器冷启动行为通过。

安全检查还包括：

- 扫描工作树和 Git 历史中的凭据形态；
- 确认 `.env`、本地数据库和密钥库数据不会被提交；
- Docker 构建上下文不包含本地秘密。

## 7. 独立评审

使用 OpenCode 执行两次相互独立的评审：

1. 规格符合性评审：对照课程要求、`SPEC.md`、`PLAN.md` 和实现逐项检查。
2. 代码质量评审：重点检查安全边界、凭据泄漏、配置兼容性、错误处理和测试缺口。

评审输入、输出摘要、人工判断和修复结论写入 `temp/`，同时在 `AGENT_LOG.md` 中留下可定位的证据索引。OpenCode 的意见必须经过验证，不会被盲目采纳。

## 8. GitHub Actions、PR 与合并

新增 GitHub Actions 工作流，至少包含：

- Python 依赖安装与单元测试；
- 演示验收；
- Docker 构建；
- 容器冷启动验证；
- 主分支上的容器镜像发布。

交付顺序：

1. 在 `setup-scaffold` 上形成职责清晰的提交；
2. 推送该分支并创建面向 `main` 的 PR；
3. 等待 GitHub Actions 真实通过；
4. 完成规格评审和代码质量评审，修复确认的问题；
5. 合并 PR；
6. 再次确认主分支工作流和镜像发布结果。

## 9. 公共容器镜像

优先发布到 GitHub Container Registry：

`ghcr.io/rippleorapple/fr-harness:latest`

工作流使用 GitHub Actions 的 `GITHUB_TOKEN` 写入 GHCR，并添加仓库来源标签。发布后验证匿名拉取。如果 GitHub 默认将包设为私有且当前令牌不能更改可见性，则保留已经成功发布的镜像和工作流证据，并准确指出唯一需要仓库所有者完成的“设为 Public”操作，不把私有镜像宣称为公共交付物。

## 10. 完成标准

只有在以下证据真实存在时才宣布对应事项完成：

- 本地全量测试与演示通过；
- Docker 构建和冷启动通过；
- OpenCode 两次评审完成，确认问题已处理或有书面判断；
- 分支已经推送，PR 已创建；
- PR 或主分支 GitHub Actions 出现真实绿色结果；
- PR 已合并；
- GHCR 镜像已发布，并经匿名拉取证明为公共；若权限阻止公开，明确标记该项未完成。

任何受外部权限限制的事项单独报告，不影响对其余已完成事项给出精确结论。
