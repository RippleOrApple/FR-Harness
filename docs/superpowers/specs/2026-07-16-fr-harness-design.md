# FR-Harness 设计规约

**日期：** 2026-07-16  
**状态：** 已完成 brainstorming，待用户审阅  
**项目类型：** AI4SE A 路线——Coding Agent Harness

## 1. 问题与目标

FR-Harness 是一个小而完整的 Python Coding Agent Harness。用户提交代码修复目标后，Harness 在受限的本地项目工作区内读取或修改文件、运行 `pytest`，并将客观测试反馈回灌给 LLM，直到任务成功、达到预算上限、被拒绝审批或失败停止。

本项目的重点不是复刻成熟编码智能体，而是用自己编写的、可确定性测试的代码实现可靠 Agent 所需要的主循环、工具分发、治理护栏、审批、测试反馈与记忆机制。

## 2. 范围

### 2.1 首版包含

- 仅支持 Python 项目，并以 `pytest` 结果作为主要客观反馈信号。
- FastAPI 极简 WebUI：创建任务、查看任务详情和审计日志、批准或拒绝危险动作。
- CLI：初始化、启动服务与运行测试等本地管理命令。
- 自行实现 Agent 主循环，使用可替换的 LLM 接口。
- MockLLM 支持离线、确定性的单元测试和机制演示。
- OpenAI 兼容 API 适配器，供用户在安全配置密钥后选择性使用真实模型。
- SQLite 持久化任务、审批、审计记录与记忆。
- Docker 分发与 CI。

### 2.2 首版不包含

- 多 Agent 编排、并发任务队列、后台 worker。
- 多语言项目支持。
- 语义向量检索或长期代码知识库。
- 自动执行未审批的危险动作。
- IDE 插件、云端多租户或生产级身份系统。

## 3. 已比较的方案

### 方案 A：轻量单进程 Harness（采用）

Python、FastAPI、SQLite 与 `pytest` 构成单进程应用；每次执行一个任务。主循环、工具分发、治理状态机和反馈闭环由项目代码实现。

优点：范围可控、机制边界清晰、可用 MockLLM 做确定性测试，适合课程 A 路线。

### 方案 B：事件驱动队列

增加后台 worker、任务队列、可恢复执行和实时日志推送。

优点是更接近生产运行时；代价是并发、失败恢复和部署会压缩核心机制的实现与测试时间，因此不在首版采用。

### 方案 C：多 Agent 分工

规划、修复和评审 Agent 分别执行。

优点是可展示编排；代价是可控性与确定性测试难度显著提高，容易退化为提示词编排，因此不在首版采用。

## 4. 架构与数据流

```text
WebUI / CLI
  → 创建修复任务
  → Agent 主循环
  → LLM 接口（MockLLM / OpenAI 兼容适配器）
  → 动作解析与工具分发
      ├─ 读文件
      ├─ 写文件
      ├─ 运行 pytest
      └─ 请求危险操作审批
  → 护栏、测试反馈、记忆与审计
  → SQLite 持久化
  → WebUI 展示状态、日志与审批项
```

任务状态机：

```text
created → running → pending_approval → running → succeeded
                                          └────→ failed
pending_approval → cancelled
running → failed
```

每个任务绑定一个工作区根目录。系统不会把工作区外的文件路径交给工具执行。

## 5. 模块与职责

| 模块 | 职责 |
| --- | --- |
| `agent` | 主循环、状态转换、上下文组装、轮数预算、停止条件 |
| `llm` | LLM 协议、MockLLM、OpenAI 兼容适配器 |
| `actions` | 结构化动作模型、动作解析与参数校验 |
| `tools` | 受限工作区读写、`pytest` 调用、工具分发 |
| `guardrails` | 路径白名单、危险操作识别、审批状态机和执行策略 |
| `feedback` | pytest 结果解析、失败摘要、反馈消息生成 |
| `memory` | 项目约定、失败尝试、审批和执行摘要的按需读取 |
| `audit` | LLM 决策、工具调用、审批、状态转换的不可变事件记录 |
| `web` | FastAPI 路由、任务/详情/审批页面 |
| `cli` | 初始化、启动服务、测试等管理命令 |

## 6. 核心机制

### 6.1 自建 Agent 主循环

`Agent.run()` 负责：

1. 读取任务、项目约定、相关失败历史和审批结果，组装上下文。
2. 调用 LLM 接口取得一个结构化动作。
3. 校验动作类型、参数和当前任务状态。
4. 经由护栏决定拒绝、待审批或允许执行。
5. 分发允许的工具调用，将结果和审计事件持久化。
6. 将测试或工具结果转为反馈，进入下一轮。
7. 在成功、失败、取消、超出最大轮数或无进展时停止。

该循环不得依赖现成 Agent 框架的高层 runner。

### 6.2 工具与动作

首版允许的结构化动作是：读取文件、写入文件、运行 pytest、请求审批与声明完成。工具实现必须验证参数，并将所有文件操作限制在任务的工作区内。

### 6.3 治理与审批（主要贡献）

危险动作默认进入 `pending_approval`。首版至少将删除文件、覆盖已有文件、执行 shell 测试命令和任何工作区外路径访问视为需要治理的操作；路径逃逸直接拒绝，其他危险动作必须由用户在 WebUI 明确批准。

审批状态机至少保证：

- `pending` 状态的操作不能执行。
- 只有指定审批项被批准后才能执行一次。
- 被拒绝、取消或过期的操作不可恢复执行。
- 每次审批决定都产生包含时间、动作、用户决定和原因的审计事件。

### 6.4 测试反馈闭环

`pytest` 的退出状态、失败测试名和精简错误摘要是确定性的外部反馈。测试失败后，Harness 将反馈写入审计和任务上下文，并要求下一轮产生新动作；它不能仅依据模型的“已修复”声明标记成功。

最大轮数或连续重复动作达到阈值时，任务失败，并记录停止原因。

### 6.5 记忆与审计

记忆保存项目约定、已失败修复尝试和审批/执行历史。每轮只注入与当前任务相关的内容，避免把所有历史内容无差别拼接进上下文。

SQLite 至少包含：`tasks`、`events`、`approvals`、`memory_entries`。审计事件覆盖 LLM 决策、护栏判定、工具调用、反馈与状态变化。

## 7. WebUI、CLI 与安全

WebUI 包含三个页面：

1. **任务页**：输入修复目标、选择受限工作区、创建任务、查看状态。
2. **任务详情页**：展示每轮动作、测试反馈、失败摘要和审计日志。
3. **审批页**：展示危险动作、原因与影响路径，让用户批准或拒绝。

CLI 至少提供初始化、启动服务和运行测试命令。

真实模型的 API Key 不得写入源码、Git 历史、日志或终端输出。首版支持通过安全引导配置；若使用 `.env`，文档必须明确其明文和进程环境可见风险。

## 8. 测试与机制演示

所有核心机制都必须在不连接网络、无需真实模型的情况下由 MockLLM 驱动测试。

必需演示：

1. **危险动作审批**：MockLLM 发起危险动作，系统进入待审批；未批准时绝不执行；批准后才继续。
2. **反馈驱动纠错**：MockLLM 首次写入错误修复，pytest 失败；Harness 回灌失败摘要；MockLLM 第二轮采取不同修复动作，pytest 通过。
3. **重点机制**：审批状态机测试，验证未批准不可执行、批准只允许指定动作执行一次、拒绝后不能执行。

额外单元测试覆盖路径逃逸、动作格式错误、非法工具参数、最大轮数、重复动作、记忆写入与审计记录。

## 9. 分发、CI 与验收

首版采用 Docker 分发；README 说明构建、运行、工作区挂载、密钥配置和已知限制。CI 在每次 push 运行单元测试并构建镜像。课程最终清单列出 `.gitlab-ci.yml` 的 `unit-test` job，同时前文提到 GitHub Actions；实际提交平台需在实现前与教师确认。

验收条件：

- MockLLM 可完整演示危险操作审批、测试反馈与修复成功。
- 一条命令运行完整测试，测试不依赖网络或真实模型。
- WebUI 可以创建任务、查看日志、批准或拒绝操作。
- 真实 LLM 可通过统一接口替换接入。
- Docker 冷启动后能按 README 运行，且不会泄露密钥。

## 10. 风险与对策

| 风险 | 对策 |
| --- | --- |
| 范围膨胀 | 明确不做多 Agent、队列和语义检索 |
| LLM 输出不合法 | 严格结构化动作解析与失败状态 |
| 文件系统越界 | 解析并校验路径位于工作区根目录内 |
| 模型自称完成但测试失败 | 只以客观测试结果决定成功 |
| 高风险操作误执行 | 默认待审批、审计与一次性批准令牌 |
| 真实密钥泄露 | 禁止硬编码/日志输出，README 给出安全配置流程 |
